# Security

Uriel assumes the worker can mutate code, run build commands, and access
repository credentials. Treat it like a privileged CI runner.

## Boundaries

- The NixOS worker owns code execution, mutable caches, job state, and
  artifacts.
- The local HTTP API is authenticated with `URIEL_WORKER_TOKEN` when configured.
- Secrets are supplied by the host environment, preferably through NixOS
  systemd environment files generated by `sops-nix`, `agenix`, or equivalent
  tooling.
- Public ingress, if needed, should be provided by a separate adapter or
  reverse proxy such as Tailscale, WireGuard, Caddy, nginx, or Hermes gateway.

## Secrets

Do not put provider secrets in Nix files or the repository. For the worker,
prefer NixOS-native secret tooling to materialize environment files under
`/run/secrets`, then pass those paths through
`services.uriel-worker.environmentFiles`.

## Repo Execution

The worker creates fresh worktrees from latest `origin/main` and never reuses
old worktrees for new jobs. This preserves target-repo worktree discipline and
keeps PRs tied to a single job.

## Artifacts

Artifacts are local files under the worker artifact directory. Do not add
commands that dump full environment variables, `.env` files, shell traces, or
secret-containing runtime config into logs, screenshots, traces, videos, or PR
comments. If an artifact needs config context, prefer allowlisting non-secret
keys.
